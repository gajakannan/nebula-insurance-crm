## Review Scope

Targeted security review of the submission intake workflow implementation and closeout evidence:

- API authorization and optimistic concurrency
- Resource-level data scoping
- Assignment and transition guard rails
- Auditability of mutations
- Data exposure on list, detail, completeness, and stale workflows

## Threat Model

| Threat | Control | Verdict |
|--------|---------|---------|
| Unauthorized user reads another team's submissions | Endpoint-level `submission:read` checks plus `SubmissionService.CanReadSubmission` and repository-side scoped query filtering | PASS |
| Distribution user or manager bypasses completeness / underwriter handoff to force `ReadyForUWReview` | `SubmissionService.TransitionAsync` calls `EvaluateCompletenessAsync` and rejects with `missing_transition_prerequisite` when required fields or documents are missing | PASS |
| Assignment endpoint used to route `ReadyForUWReview` work to a non-underwriter or out-of-scope user | `SubmissionService.AssignAsync` validates assignee existence, active status, role, and requester scope via `CanAssignSubmission` | PASS |
| Mutations occur without optimistic concurrency, enabling replay or stale updates | Update, transition, and assignment endpoints require `If-Match`; stale versions return `precondition_failed` | PASS |
| Timeline actor or transition history can be spoofed or omitted | Mutation paths stamp `ActorUserId` from the authenticated principal and append immutable timeline / transition records | PASS |
| Stale dashboard or list counts leak submissions outside the caller's scope | `SubmissionRepository.BuildStaleFlagsAsync` operates on already scoped candidate sets produced from the caller's ABAC-filtered query | PASS |

## Verified Controls

### Authorization

- `engine/src/Nebula.Api/Endpoints/SubmissionEndpoints.cs`
  - Endpoint group requires authorization.
  - Each handler checks Casbin action access before invoking the service layer.
- `engine/src/Nebula.Application/Services/SubmissionService.cs`
  - `CanReadSubmission` enforces assignee, region, managed-broker, and managed-program scope.
  - `CanAssignSubmission` limits reassignment to Admin, in-region DistributionManager, or the current DistributionUser owner.
  - `CanPerformTransition` keeps intake transitions with distribution roles and reserves downstream transitions for Underwriter/Admin.

### Input Validation and Concurrency

- Create, update, transition, and assignment requests all pass through FluentValidation and endpoint-level parsing.
- `TryParseExpectedRowVersion` enforces `If-Match` on update, transition, and assignment requests.
- `SubmissionService` rechecks `RowVersion` before mutation and catches `DbUpdateConcurrencyException` as `precondition_failed`.

### Auditability

- Create, update, assignment, and transition flows each append `ActivityTimelineEvent` records.
- Transition flows also append `WorkflowTransition` history entries.
- Event payloads record before/after values or actor metadata without trusting client-supplied actor identity.
