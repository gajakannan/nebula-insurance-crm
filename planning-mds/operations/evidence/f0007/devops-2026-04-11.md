## Migration Safety Assessment

### Forward Path (Up)

| Concern | Assessment |
|---------|------------|
| **Data loss** | No data loss. Old `RenewalDate` is converted to `PolicyExpirationDate` and `TargetOutreachDate` before the column is dropped. Status values are mapped to the new 6-state model. | 
| **Non-nullable columns** | PolicyId, PolicyExpirationDate, TargetOutreachDate added as nullable, backfilled from existing data, then altered to non-nullable. This is the correct three-step pattern for existing data. |
| **Orphaned FK references** | Policy stub migration backfills `Policies` rows for any `Renewals.PolicyId` or `Renewals.BoundPolicyId` that don't already have a matching Policy record. Uses `DISTINCT ON` to handle duplicates. |
| **Index safety** | Filtered unique index `IX_Renewals_PolicyId_Active` uses `WHERE IsDeleted=false AND CurrentStatus NOT IN ('Completed','Lost')` — safe for concurrent reads. Partial index `IX_Renewals_TargetOutreachDate` only indexes Identified-status rows. Expression index on `WorkflowSlaThresholds` uses `COALESCE(LineOfBusiness, '__default__')` to maintain uniqueness with nullable LOB. |
| **Seed data** | 6 `ReferenceRenewalStatus` rows replace 15 old rows. 5 per-LOB `WorkflowSlaThreshold` rows inserted. Fixed GUIDs used for idempotent re-application. |
| **Lock duration** | Status mapping UPDATE and reference data DELETE/INSERT operate on small row counts in dev/test. In production with existing data, the `UPDATE Renewals SET CurrentStatus = CASE...` would be a full-table scan — acceptable for a one-time migration; table lock duration is proportional to row count. |

### Backward Path (Down)

| Migration | Down Coverage |
|-----------|--------------|
| `F0007_ReconcileRenewalWorkflowStates` | Reverse CASE mapping covers all 6 new statuses back to a reasonable old status. ReferenceRenewalStatuses restored with original 15 rows (fixed GUIDs). |
| `F0007_ReconcileRenewalEntityShape` | New columns dropped. RenewalDate restored from PolicyExpirationDate (converted back to timestamp with time zone). SubmissionId name restored. Old indexes recreated. FK to Submissions restored. |
| `F0018_PolicyStubAndF0007RenewalSlaReconcile` | Policies table dropped entirely. FK from Renewals to Policies dropped first. Expression index dropped via raw SQL. Per-LOB threshold rows deleted by fixed GUID. LineOfBusiness column dropped from WorkflowSlaThresholds. Old unique index restored. |

**Rollback is complete and non-destructive** for a clean dev environment. In production with data, the rollback would lose Policy stub records, but since F0018 hasn't shipped production data yet, this is acceptable.

## Deployment Considerations

1. Migrations must be applied in order: ReconcileRenewalWorkflowStates → ReconcileRenewalEntityShape → PolicyStubAndRenewalSlaReconcile.
2. The status remapping migration should run during a maintenance window if production data exists, as the full-table UPDATE on Renewals will hold a row-level lock per row.
3. After migration, the old 15 renewal statuses will no longer be valid — any external integrations referencing old status codes must be updated.

All three migrations have complete forward and backward paths, correct data backfill logic, and safe index creation patterns. No new infrastructure requirements. PASS.
