## Threat Model

F0004 introduces cross-user task assignment, creator-based access, and a user search endpoint. Key threat vectors:

1. **Privilege escalation** — Non-manager creates/assigns tasks to others
2. **Horizontal access** — User accesses tasks they don't own or didn't create
3. **IDOR** — Entity existence leakage via distinct 403 vs 404 responses
4. **User enumeration** — User search endpoint leaks user data to unauthorized roles
5. **Status manipulation** — Non-assignee changes task status

## IDOR Prevention

| Scenario | Expected Response | Actual |
|----------|------------------|--------|
| GET /tasks/{id} — task exists, no access | 404 (not_found) | 404 |
| GET /tasks/{id} — task does not exist | 404 (not_found) | 404 |
| PUT /tasks/{id} — task exists, no access | 404 (not_found) | 404 |
| DELETE /tasks/{id} — task exists, no access | 404 (not_found) | 404 |

Both not-found and not-authorized normalize to 404 `not_found` to prevent entity existence leakage.

## Application-Layer Guards (Defense in Depth)

These guards operate inside `TaskService` independent of Casbin:

### Status-change restriction
- **Guard:** `if (body.Status is not null && task.AssignedToUserId != userId) → 403 status_change_restricted`
- **Tests:** `UpdateAsync_CreatorCannotChangeStatus_ReturnsStatusChangeRestricted`, `UpdateTask_CreatorChangesStatus_Returns403StatusChangeRestricted`
- **Assessment:** PASS — Creator who is not assignee cannot change status, even though Casbin grants update access

### Reassignment restriction
- **Guard:** `if (body.AssignedToUserId is not null && task.CreatedByUserId != userId) → 403 not_found`
- **Tests:** `UpdateAsync_AssigneeCannotReassign_ReturnsForbidden`
- **Assessment:** PASS — Assignee who is not creator cannot reassign

### Assignee validation
- **Guard:** `if (assignee is null) → 422 invalid_assignee; if (!assignee.IsActive) → 422 inactive_assignee`
- **Tests:** `CreateAsync_AssignToNonExistent_ReturnsInvalidAssignee`, `CreateAsync_AssignToInactiveUser_ReturnsInactiveAssignee`
- **Assessment:** PASS — Prevents assignment to non-existent or deactivated users

### Non-manager cross-assignment guard
- **Guard:** `if (!isManager && body.AssignedToUserId != userId) → 403 not_found`
- **Tests:** `CreateAsync_NonManagerAssignsToOther_ReturnsForbidden`, `CreateTask_NonManagerAssignsToOther_Returns403`
- **Assessment:** PASS — Only DistributionManager + Admin can assign to others

## Frontend Authorization Alignment

### MANAGER_ROLES correction (DEF-03)
During review, `MANAGER_ROLES` in 3 frontend files included `ProgramManager` and `RelationshipManager`, which would have shown the "Assigned By Me" tab and cross-user assignment picker to unauthorized roles. **Fixed** to `['DistributionManager', 'Admin']` only.

**Impact if unfixed:** Frontend UX leakage only — backend correctly rejects unauthorized requests (403). No privilege escalation possible, but unauthorized users would see UI elements that always fail.

### Status button visibility correction (DEF-04)
Status action buttons were visible to creators who are not assignees. **Fixed** — only assignees see status buttons.

**Impact if unfixed:** Frontend UX leakage only — backend correctly rejects status changes from non-assignees (403).
