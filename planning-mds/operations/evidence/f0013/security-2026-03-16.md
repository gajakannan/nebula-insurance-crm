## 3) Threat model focus (F0013-specific)

Assets:
- Opportunity aggregation data (`/dashboard/opportunities/*`) including broker/program/LOB/state grouping.
- LOB distribution and SLA telemetry, which can reveal portfolio composition and operational performance.

Actors:
- Internal roles (DistributionUser, DistributionManager, Underwriter, RelationshipManager, ProgramManager, Admin).
- BrokerUser (explicitly constrained for dashboard aggregate resources).

Trust boundaries:
- Browser → API (`/dashboard` routes).
- API authorization guard (Casbin role checks) → repository query layer (scope filters expected by policy docs).
- API response boundary exposing aggregated business data.

## 5) Vulnerability findings

### High

1. Missing query-layer scope enforcement on dashboard aggregate/breakdown reads for internal non-admin roles.
   - CWE: CWE-284 (Improper Access Control), CWE-200 (Exposure of Sensitive Information)
   - Locations:
     - `engine/src/Nebula.Api/Endpoints/DashboardEndpoints.cs:61`
     - `engine/src/Nebula.Api/Endpoints/DashboardEndpoints.cs:104`
     - `engine/src/Nebula.Application/Services/DashboardService.cs:15`
     - `engine/src/Nebula.Application/Services/DashboardService.cs:24`
     - `engine/src/Nebula.Infrastructure/Repositories/DashboardRepository.cs:69`
     - `engine/src/Nebula.Infrastructure/Repositories/DashboardRepository.cs:331`
     - Policy contract requiring query-layer scope: `planning-mds/security/policies/policy.csv:157` and `planning-mds/security/policies/policy.csv:178`
   - Evidence:
     - Endpoint checks only role/resource allow (`dashboard_pipeline`) and then executes repository reads without user/scope parameters.
     - Repository dashboard queries aggregate across full datasets (`Submissions`, `Renewals`, joins to `Brokers`/`Programs`) with no role-specific filters.
   - Exploit scenario:
     - A `DistributionUser` with legitimate dashboard access can retrieve cross-region, cross-assignment opportunity composition (including broker and LOB breakdown) outside intended scope by querying breakdown endpoints directly.
   - Required remediation:
     1. Thread `ICurrentUserService` (or an explicit `DashboardScope`) into dashboard service/repository methods.
     2. Apply role-specific scope filters in every dashboard read query (`opportunities`, `flow`, `aging`, `hierarchy`, `outcomes`, `breakdown`) before aggregation.
     3. Add integration tests proving cross-scope denial/filtered results for each internal role class, not only BrokerUser.
     4. Add negative tests for breakdown `groupBy=broker|lineOfBusiness|brokerState` to ensure no out-of-scope counts leak.

### Medium

1. Security scanning gate is incomplete in current environment.
   - Locations:
     - `agents/security/scripts/check-secrets.sh` (gitleaks missing)
     - `agents/security/scripts/run-sast-scan.sh` (semgrep missing)
     - `agents/security/scripts/run-dast-scan.sh` (no target URL configured)
     - `agents/security/scripts/scan-dependencies.sh` (network/DNS failure + backend restore-scan failure)
   - Risk:
     - Unknown residual vulnerabilities in dependencies and code paths due unavailable scanners.
   - Required remediation:
     1. Install required scanner toolchain in CI/runtime image (`gitleaks`, `semgrep`).
     2. Configure deterministic DAST target for review gates.
     3. Run dependency scan in network-enabled CI and attach output artifact to evidence.
